# Releasing Commons

How the five publishable packages get versioned and published to npm. If you
only want to *propose* a release, you just need the [Add a changeset](#1-add-a-changeset)
step ‚Äî the rest is automated.

The packages:

| Package | npm name |
| --- | --- |
| Tokens | `@21stgov/commons-tokens` |
| Core | `@21stgov/commons-core` |
| React | `@21stgov/commons-react` |
| Fonts | `@21stgov/commons-fonts` |
| CLI | `@21stgov/commons` |

The `docs`, `playground`, and `registry` apps are `private` and never published.

## Where we publish

Commons publishes to the **public npm registry** ([npmjs.com](https://www.npmjs.com)),
under the `@21stgov` scope ‚Äî **not** GitHub Packages.

Why: anyone should be able to run `npx @21stgov/commons add button` or
`npm install @21stgov/commons-react` with zero setup. GitHub Packages' npm
registry requires every consumer to authenticate with a personal access token,
even for public packages ‚Äî a non-starter for a public design system. So the
scope lives on npmjs.com and installs need no auth.

One-time org setup (already done if `npm view @21stgov/commons` resolves):
create the **`@21stgov` organization on npmjs.com** and make the scope public.

## Versioning

- **[Semantic versioning](https://semver.org).** We are **pre-1.0** ‚Äî every
  component is `experimental`, and `0.x` **minor** releases may include breaking
  changes; **patch** releases are fixes only.
- **Lockstep.** All five packages ship the same version, bumped together, so a
  consumer never has to reason about cross-package compatibility. We keep them
  in sync by listing every package in the release changeset ‚Äî **not** via
  Changesets' `fixed` group (with `0.0.x` packages that group forces a jump to
  `1.0.0` on the first minor, which we don't want pre-1.0).
- **The road to 1.0** runs through the accessibility work that isn't finished:
  manual screen-reader and inclusive-user testing, tracked per component in the
  `screenReadersTested` field of its registry contract. 1.0 is when the core set
  is `stable` and the public API has settled.

## The normal flow (automated)

We use [Changesets](https://github.com/changesets/changesets). A GitHub Action
([`.github/workflows/release.yml`](.github/workflows/release.yml)) does the
version bump and the publish; you write changesets and merge PRs.

### 1. Add a changeset

With any change that should ship, from the repo root:

```sh
pnpm changeset
```

Pick the affected packages, the bump type (`patch` / `minor`), and write a
one-line summary ‚Äî it becomes the CHANGELOG entry. **For a coordinated release,
select all five packages with the same bump type** (lockstep). Commit the
generated `.changeset/*.md` file with your change.

> A change that doesn't affect any published package (docs copy, a demo, CI)
> needs no changeset.

> **Also update the curated changelog.** The repo-root
> [`CHANGELOG.md`](CHANGELOG.md) is the single, human-facing history for the
> whole system (the per-package `CHANGELOG.md` files are machine-generated by
> Changesets and are an implementation detail). Add a curated entry for the new
> version there ‚Äî the docs site's `/docs/changelog` page is generated from it,
> so you only edit the one file.

### 2. Merge the "Version Packages" PR

When changesets land on `main`, the Release workflow opens (or updates) a
**Version Packages** PR that applies them ‚Äî bumping every selected package's
version and appending to its `CHANGELOG.md`. Review it like any PR.

### 3. Publish happens on merge

Merging the Version Packages PR leaves no changesets on `main`, so the next run
of the Release workflow **publishes the changed packages to npm** (with
provenance) and tags the release. Nothing else to do.

### One-time repo setup

Add a single repository secret:

- **`NPM_TOKEN`** ‚Äî an npm **Automation** access token for the `@21stgov` org
  (npmjs.com ‚Üí *Access Tokens* ‚Üí *Generate New Token* ‚Üí *Automation*). Put it in
  *Settings ‚Üí Secrets and variables ‚Üí Actions*.

`GITHUB_TOKEN` is provided automatically. The workflow already has the
`id-token: write` permission for npm provenance.

## Manual release (fallback)

If you ever need to publish by hand (CI down, a one-off), from a clean `main`
with your changesets committed:

```sh
pnpm install
pnpm build                 # every package needs its dist/
pnpm changeset version     # apply changesets ‚Üí bump versions + CHANGELOGs
git commit -am "chore: release packages"
pnpm changeset publish     # publish changed packages to npm (respects access: public)
git push --follow-tags
```

You must be logged in to npm (`npm whoami`) as a member of the `@21stgov` org
with publish rights.

## Pre-flight checklist

Before a release goes out, the full gate should be green (CI enforces this on
every PR):

```sh
pnpm build
pnpm test            # 1600+ unit/axe tests + registry contract + contrast audit
pnpm typecheck
pnpm lint
pnpm validate:contrast
```

All five packages already declare `"publishConfig": { "access": "public" }` and
`"engines": { "node": ">=22" }`; the `consumer-node-minimum` CI job packs real
tarballs and smoke-tests them on Node 22, so a broken published artifact fails
CI before it can ship.
