---
title: "MCP server"
description: "A local, read-only MCP server built into the Commons CLI — let your coding agent search, inspect, and plan Commons installs safely."
---

Commons ships a [Model Context Protocol](https://modelcontextprotocol.io)
server inside the CLI package. `commons mcp` starts a **local, read-only**
stdio server — no separate install, no API key, no telemetry. It gives MCP
clients like Claude Code, Cursor, VS Code, and Codex a structured way to
discover Commons components, read their contracts, and plan installations
against your actual project.

Read-only is the design, not a limitation: the server never writes to your
project. Installation stays in the CLI, where the plan is previewable and the
outcome is atomic — the model returns you the exact command to run.

## Setup

One command writes the client configuration for you:

<Tabs groupId="package-manager" persist items={["npm", "pnpm", "yarn", "bun"]}>

<Tab value="npm">

```sh
npx @21stgov/commons mcp init --client claude
```

</Tab>

<Tab value="pnpm">

```sh
pnpm dlx @21stgov/commons mcp init --client claude
```

</Tab>

<Tab value="yarn">

```sh
yarn dlx @21stgov/commons mcp init --client claude
```

</Tab>

<Tab value="bun">

```sh
bunx @21stgov/commons mcp init --client claude
```

</Tab>

</Tabs>

`--client claude` writes a project-scoped `.mcp.json`:

```json
{
  "mcpServers": {
    "commons": {
      "command": "npx",
      "args": ["@21stgov/commons", "mcp"]
    }
  }
}
```

- `--client cursor` writes the same shape to `.cursor/mcp.json`.
- `--client vscode` writes `.vscode/mcp.json`, with the top-level key
  `servers` instead of `mcpServers` (VS Code's format).
- `--client codex` prints a snippet to add to `~/.codex/config.toml`, since
  Codex configuration is user-level TOML:

```toml
[mcp_servers.commons]
command = "npx"
args = ["@21stgov/commons", "mcp"]
```

There are no secrets in any of these files — the server is local and
read-only — so they are safe to commit and share with your team.

## Tools

The tool surface is small and bounded on purpose:

| Tool | Input | Returns |
| ---- | ----- | ------- |
| `search_components` | `{ query, limit? }` | Matching catalog entries — name, status, description, use-when guidance |
| `get_component` | `{ name }` | The component's full registry-item v1 JSON: files, dependencies, the [accessibility contract](/docs/accessibility), and the framework-agnostic `.cui-*` HTML markup (`html`) for the non-React path |
| `plan_install` | `{ names, cwd? }` | A dry-run install plan: every file with its `write` / `conflict` / `skip` action, npm dependencies, and the exact `installCommand` to run. **Never writes.** |
| `get_setup` | — | The registry URL, the five CSS imports in order, and tsconfig alias guidance |

`plan_install` is the same pre-flight the CLI's `add --dry-run` performs, so
what the agent shows you is exactly what `add` will do — including conflicts
with files you have already modified, surfaced before anything is written.

## Prompts that work well

- "What Commons components exist for collecting a date?"
- "Plan installing button and field into this project."
- "Show me the accessibility contract for radio-group before we use it."
- "Give me the `.cui-*` HTML markup for an alert — I'm not using React."
- "What setup does this project still need for Commons?"

A good agent workflow is search → inspect → plan → hand you the `add`
command. The server gives the model versioned facts from the registry instead
of trained guesses, and the write stays a human decision.

## How it resolves the registry

The MCP server reads `commons.json` and the `COMMONS_REGISTRY` environment
variable exactly like every other CLI command. If your project points at a
[self-hosted registry](/docs/cli#self-hosted-registries), the agent searches
and plans against that registry — same code path, same answers.

## Hosted server (no install)

There is also a **hosted, read-only** MCP server so any agent can discover and
plan Commons installs with **zero setup** — no CLI, no project on disk:

| Transport | URL |
| --- | --- |
| Streamable HTTP | `https://mcp.commonsui.com/mcp` |
| SSE (legacy) | `https://mcp.commonsui.com/sse` |

Add it to a client that supports remote MCP servers — for example:

```sh
claude mcp add --transport http commons https://mcp.commonsui.com/mcp
```

or, in a client's JSON config:

```json
{
  "mcpServers": {
    "commons": { "url": "https://mcp.commonsui.com/mcp" }
  }
}
```

Any MCP-capable client connects the same way — Claude Code, Cursor, VS Code,
Cline, Continue, Windsurf, and Zed all accept a remote MCP URL (check your
client's docs for whether the key is `mcpServers`, `servers`, or a UI field).

It exposes the same four tools, resolved against the public registry
(`commonsui.com/r`). The difference from the local server: with no project on
disk it can't read your `commons.json` or detect file conflicts, so
`plan_install` reports the registry file paths and hands you the exact
`npx @21stgov/commons add …` command to run locally.

**Use the local server** (above) to plan against *your* project and install;
**use the hosted server** to let any agent explore Commons without installing
anything.

## Agents without MCP (llms.txt)

For tools that ingest plain text rather than speak MCP — ChatGPT, local models
run through Ollama or LM Studio, or any assistant with a "read this URL" step —
Commons publishes its full documentation as flat text:

| File | Contents |
| --- | --- |
| [`commonsui.com/llms.txt`](https://commonsui.com/llms.txt) | A concise index of the docs, following the [llms.txt](https://llmstxt.org) convention |
| [`commonsui.com/llms-full.txt`](https://commonsui.com/llms-full.txt) | Every doc page inlined — paste it in when a model needs the complete context |

## Give your project's agent Commons context

Drop a short brief into your project's agent instructions — `AGENTS.md`,
`CLAUDE.md`, `.cursorrules`, `.github/copilot-instructions.md`, or your client's
rules file — so any agent working in your repo uses Commons correctly:

```md
This project uses Commons (@21stgov/commons), an accessibility-first design
system for U.S. local governments.

- Add components with the CLI, never by hand-copying from the docs:
  `npx @21stgov/commons add <component>`. You own the copied source.
- Discover and inspect through the Commons MCP server (search_components,
  get_component, plan_install) when available, or `npx @21stgov/commons search`
  / `inspect`.
- Each component's accessibility contract is NORMATIVE: preserve keyboard
  behavior, roles, accessible names, focus management, and 44px targets. Never
  strip ARIA wiring or focus rings.
- Style with the `--cui-*` design tokens; never hardcode hex colors. Switch
  themes with `data-theme` on the root element.
- No React? Use `commons.css` (the `.cui-*` classes) plus the
  `@21stgov/commons-js` runtime; copy markup from a component's HTML → Code tab.
- Full docs as text: https://commonsui.com/llms.txt
```

Claude users can go further and install the **Commons skill**, which packages
this guidance plus the install workflow — see
[`skills/commons-ui`](https://github.com/21stgov/commons/tree/main/skills/commons-ui)
in the repository.

## Notes

- **The local server is local by design.** It runs on your machine over stdio
  and sends nothing anywhere. No telemetry, no analytics, no model calls inside
  the tools. The hosted server is read-only and fetches only the public
  registry.
- **Read-only by design.** There is no tool that writes files. If a model
  claims it installed a component through MCP, it did not — check for the
  CLI command it should have given you instead.
- **Both paths are covered.** `get_component` returns both the React `files`
  and the framework-agnostic `.cui-*` `html`, so agents building either way get
  correct markup. The install-oriented tools (`plan_install`, `get_setup`) target
  the React own-your-code flow; on the HTML path there is nothing to install —
  load `commons.css` + `commons.js` ([Without React](/docs/without-react)) and
  use the `html` markup.
